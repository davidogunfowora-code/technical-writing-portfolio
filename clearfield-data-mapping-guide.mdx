## 1. Purpose and Scope

This guide defines the standard schema used by the Clearfield Analytics Platform for all client data ingestion. Every client submits data in a proprietary format with field names, data types, and structural conventions unique to their source system. This document defines the target schema — the standard field names, data types, and constraints that all client data must conform to before it can be loaded to Snowflake.

The data mapping process translates a client's native field names to the Clearfield standard field names defined in this guide. Each client maintains a client-level mapping document that records the specific translation rules applied to their files. This guide is the master reference that all client mapping documents are built against.

## 2. Schema Standards — Naming Conventions

All field names in the Clearfield standard schema follow the conventions below. Client mapping documents must translate all native field names to these conventions before ingestion.

### 2.1 General Naming Rules

- All field names are in `UPPERCASE` with no spaces
- Multi-word field names use no separator (e.g., `GIFTAMOUNT`, not `GIFT_AMOUNT` or `GiftAmount`)
- Prefixes identify the table the field belongs to (e.g., `GIFT` for transaction fields, `DONOR` for donor fields)
- PII fields are identified in the field tables below and routed to `PII_SECURE` at ingestion

### 2.2 Table Prefixes

| Prefix | Table | Description |
|---|---|---|
| (none) | DIMDONOR | Donor dimension attributes |
| GIFT | FACTGIFT | Gift transaction fields |
| PROMO | FACTPROMOTION | Promotion / mail file fields |
| DONORHIST | FACTDONORHISTORY | Computed annual history fields |

> DONORID must be identical in format across the Donor File, Transaction File, and Promotion File. Mixed formats (e.g., integer in one file, padded string in another) will cause join failures and result in unmatched gift and promotion records. Verify format consistency before submitting files.

## 4. Transaction (Gift) File — Field Definitions

**Target table:** `FACTGIFT`
**One row per gift transaction.**

|---|---|---|---|---|
| GIFTID | VARCHAR | Required | No | Unique gift/transaction identifier. Deduplication key. |
| DONORID | VARCHAR | Required | No | Donor identifier. Foreign key to DIMDONOR. Must match format exactly. |
| GIFTDATE | DATE | Required | No | Date gift was received. Format: YYYY-MM-DD. Used for FY assignment and trend analysis. |
| GIFTAMOUNT | DECIMAL(10,2) | Required | No | Dollar value of gift. Core revenue metric. |
| SOURCECODE | VARCHAR | Required | No | Campaign attribution source code. Primary join key to Source Code Lookup Table. |
| GIFTTYPE | VARCHAR | Required | No | Payment method: Check, Credit Card, EFT, PayPal, Pledge, Stock, Other. |
| GIFTRESPONSECHANNEL | VARCHAR | Required | No | Response channel: Mail, Online, Phone, Walk-in, Other. Used for matchback attribution. |
| RECURRINGGIFTFREQ | VARCHAR | Required | No | Recurring gift frequency: Monthly, Quarterly, Annually, One-Time, None. |
| GIFTPAYMENTMETHOD | VARCHAR | Optional | No | Specific payment method detail (e.g., Visa, MasterCard, ACH). |
| PLEDGETYPE | VARCHAR | Optional | No | If gift type is Pledge: pledge details or installment plan. |
| RECURRINGGIFTSTATUS | VARCHAR | Optional | No | Active sustainer status: Active, Cancelled, Suspended. |
| RECURRINGGIFTSTART | DATE | Optional | No | Date recurring gift program began. Format: YYYY-MM-DD. |
| RECURRINGGIFTEND | DATE | Optional | No | Date recurring gift cancelled or ended. Format: YYYY-MM-DD. |
| FY | INTEGER | Computed | No | Fiscal year of gift. Computed from GIFTDATE and client FY definition at ingestion. |

> **SOURCE CODE INTEGRITY**
> A gift without a valid SOURCECODE cannot be attributed to any campaign and will appear as unattributed revenue in the Campaign Performance report. Source codes must match the format defined in the Source Code Structure Reference Guide exactly — including case, spacing, and delimiters.

## 6. PII Flagging Standards

All fields marked **YES — PII_SECURE** in the field tables above are subject to the following handling rules:

1. **PII fields are never stored in the analytics tables** (DIMDONOR, FACTGIFT, FACTPROMOTION). They exist in the source file for ingestion processing only.
2. **At ingestion**, all PII fields are extracted from the source file and written exclusively to the `PII_SECURE` table in Snowflake.
3. **PII_SECURE is encrypted at rest** and access is restricted to authorized roles only. Standard analytics service accounts do not have access to PII_SECURE.
4. **DONORID is the only link** between PII_SECURE and the analytics tables. No other PII field is present in any analytics table.
5. **Acquisition files** (Promotion Files containing prospect records not yet in the client's donor database) contain PII because matchback analysis requires name-and-address matching. This PII is still routed to PII_SECURE at ingestion.

| PII Field | Present In | Reason |
|---|---|---|
| FIRSTNAME, LASTNAME | Donor File, Promotion File (ACQ) | Identity record, matchback |
| STREETADDRESS, CITY, ZIPCODE | Donor File, Promotion File (ACQ) | Mail delivery, matchback analysis |
| EMAILADDRESS | Donor File | Email solicitation, digital attribution |
| PHONE | Donor File | Phone solicitation |

## 7. Validation Rules & Ingestion Requirements

All files must pass the following validation checks before ingestion. Files failing any Required validation will be rejected and returned to the submitting team.

| Rule | Applies To | Required? | Description |
|---|---|---|---|
| DONORID format consistency | All three files | Required | DONORID must be the same format (length, type, padding) across Donor, Transaction, and Promotion files. |
| No null DONORID | All three files | Required | No record may have a null or blank DONORID. Records with null DONORID will be rejected. |
| No null SOURCECODE | Transaction File, Promotion File | Required | SOURCECODE must be present on every row. Null source codes cannot be attributed. |
| GIFTDATE format | Transaction File | Required | All dates in YYYY-MM-DD format. Mixed formats will cause ingestion failure. |
| MEDIAEVENTDATE present | Promotion File | Required | Every promotion record must have a mail date. Records without mail dates cannot establish a response window. |
| FY consistency | Promotion File | Required | The FY field must match the fiscal year encoded in the SOURCECODE for that record. |
| MAILQUANTITY > 0 | Promotion File | Required | Mail quantity must be a positive integer. Zero or negative values indicate a data error. |
| No duplicate GIFTID | Transaction File | Required | GIFTID must be unique across all transaction records for a given client. |
| No duplicate PROMOID | Promotion File | Required | PROMOID must be unique per promotion record submission. |
| PII fields present for ACQ | Promotion File (ACQ) | Required for matchback | Acquisition promotion files must include FIRSTNAME, LASTNAME, STREETADDRESS, and ZIPCODE for matchback analysis. |

## 8. Client Mapping Checklist

Before submitting a new client mapping document for review, confirm that all items below are complete.

- [ ] All three file types (Donor, Transaction, Promotion) have been reviewed against the field tables in Sections 3–5
- [ ] Every required field has a mapping from the client's native field name to the Clearfield standard field name
- [ ] DONORID format has been confirmed consistent across all three files
- [ ] All PII fields have been identified and marked for PII_SECURE routing
- [ ] Date fields have been confirmed in YYYY-MM-DD format (or a conversion has been specified)
- [ ] SOURCECODE format has been validated against the Source Code Structure Reference Guide
- [ ] Fiscal year definition for this client has been documented and applied to FY field calculation logic
- [ ] Validation rules in Section 7 have been reviewed and confirmed against a sample file
- [ ] Any fields that do not have a direct mapping have been documented with a transformation rule or flagged for analytics team review
